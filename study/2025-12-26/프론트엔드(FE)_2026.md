# 프론트엔드 2026

2025 키워드 → AI, 프론트엔드 개발자의 직무 범위 확장

State of JS 2025 설문조사, 원티드 데이터 → 프론트엔드 채용 공고의 40%이상이 CI/CD, Docker, AWS/Vercel 과 같은 인프라 운영 능력을 요구

AI가 implementation 영역을 대체함에 따라, 전체 시스템을 조망하는 아키텍처 설계 능력 요구됨

---

## 1. 프레임워크

### 1) React 19, 서버 컴포넌트 가속화

**클라이언트 중심에서 서버중심(Server-First)로 이동**

**서버 컴포넌트(React Server Component, RSC)와 Actions**

- useEffect를 사용해서 데이터 페칭을 하면서 번들 사이즈로 고민했다면, REACT 19에서는 백엔드 로직을 컴포넌트 내부로 직접 가져올 수 있음.
- 벤치마크 결과 → RSC 도입한 이커머스 프로젝트의 경우 TTI(Time to Interactive)가 평균 30% 이상 단축, 클라이언트 번들사이즈 최대 45%까지 감소
- 클라이언트 상태 관리 라이브러리(ex. TanStack Query)의 역할이 축소되고, 대신 서버에서 미리 렌더링된 HTML을 스트리밍으로 받아오는 패턴이 표준이 되고 있음 → 모든 컴포넌트는 기본적으로 서버컴포넌트이며 상호작용이 필요한 경우에만 클라이언트로 전환(”use client”) 원칙

---

### 2) Next.js 15, 16

**터보팩(Turbopack)**

- 자바스크립트와 타입스크립트에 최적화된 번들러
    - 번들러: 웹 애플리케이션을 구성하는 여러 파일들(JavaScript, CSS, 이미지 등)을 브라우저에서 실행가능한 최적화된 파일로 변환하는 도구
- Next 15는 Rust 기반 터보팩을 도입해 빌드와 HMR 속도 올리고, 기존 웹팩 대비 콜드스타트 시간을 53% 단축

---

### 3) 타입스크립트와 런타임 안정성

- 컴파일 타임을 넘어 런타임까지
    - 최근 업데이트된 satisfies 연산자와 성능 최적화
    - tRPC 또는 GraphQL을 활용해 백엔드 스키마 변경이 프론트엔드 빌드 에러로 즉시 감지되는 E2E 타입 안정성 환경 구축이 필수

## 2. 아키텍처, 인프라

### 1) 엣지 컴퓨팅(Edge Computing)

- Cloudflare Workers나 Vercel Edge Middleware를 활용하여 사용자의 요청이 오리진 서버에 도달하기 전에 처리하는 로직
    - 사용자 국가 식별한 통화 변환, 엣지단 A/B 테스트 그룹 분기 처리
- ISR(Incremental Static Regeneration)과 온디맨드 재검증 결합
    - 쇼핑몰 상품 정보처럼 변경 잦은 데이터도 전체 빌드 없이 특정 페이지만 엣지 노드에서 갱신하는 전략을 통해 정적 사이트(Static Site)의 속도와 동적 사이트의 최신성 동시 확보
    
    → 프론트엔드: HTTP 헤더, CDN 전략 수립
    

---

### 2) Rust번들러 기반의 마이크로 프론트엔드 고도화

- 모듈 페더레이션 기술 안정화
    - MFE(Micro-Frontend): MSA(Micro Service Architecture)를 프론트엔드에 적용한 것
    - MFE(Micro-Frontend) 구현 방식 중 하나로 모듈 페더레이션 키워드 등장
    - **Module Federation**: 하나의 앱을 독립적인 배포가 가능한 모듈 단위로 나누어 브라우저 런타임에 합체시키는 개념
        - 빌드 타임에 통합하면 A 애플리케이션 모듈이 수정될 때, 이를 사용하던 다른 애플리케이션도 다시 빌드 및 배포해야해서 MFE의 장점인 독립적 배포가 어려워짐
- 마이크로 프론트엔드 구조
    - 내 코드가 다른 팀의 서비스에 미칠 영향 고려해야함
    - 공통 디자인 시스템과 상태 공유 전략을 수립하는 고차원적인 아키텍처 설계 역량 요구됨

---

### 3) API 스펙 동기화: BFF(Backend for Frontend)

- tRPC, Hono와 같은 RPC(Remote Procedure Call) 스타일의 프레임워크
    - 백엔드 함수 하나가 곧 프론트엔드 API 호출 함수가 되는 구조

### 2) 디자인-코드 통합의 자동화(Design-to-Code)

- **Figma Dev Mode, v0**
    - 퍼블리싱 업무의80% 자동화
    - 단순히 HTML/CSS를 그려주는 수준을 넘어, 구체적인 기술스택 명시하면 사용 가능한 수준의 코드를 생성해줌 → 프론트엔드 개발자: 복잡한 클라이언트 상태관리, 성능 최적화에 집중
    - 디자이너가 만든 시안을 AI가 1차 코드로 변환, 개발자는 로직만 연결하여 배포하는 무손실(No-handoff) 워크플로우 확산
