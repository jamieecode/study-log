# 동시성 제어(Concurrency Control) 방식

여러 사용자가 동시에 같은 자원에 접근할 때 데이터 정합성을 보장하기 위해 다양한 **락(Lock) 전략**이 사용된다. 상황에 따라 DB 레벨, 애플리케이션 레벨, 외부 인프라 레벨의 락을 선택해야 한다.

---

## 1. 비관적 락 (Pessimistic Lock)

충돌이 **발생할 것을 가정**하고, 데이터를 조회하는 시점부터 락을 걸어 다른 트랜잭션의 접근을 차단하는 방식.

### 구현 방식

```sql
SELECT * FROM table WHERE id=1 FOR UPDATE;
```

### 특징

- 트랜잭션이 끝날 때까지 다른 트랜잭션의 수정 불가
- DB 레벨에서 직접 제어
- 재고 차감, 좌석 예약 등 **충돌 가능성이 매우 높은 쓰기 작업**에 적합

### 장점

- 데이터 정합성이 매우 중요한 경우 안전함
- 충돌이 빈번한 환경에서 확실한 동시성 제어 가능

### 단점

- 락 대기로 인해 응답 지연 발생 가능
- 트래픽이 많을 경우 DB 커넥션 병목 발생
- 데드락(Deadlock) 위험 존재

---

## 2. 낙관적 락 (Optimistic Lock)

충돌이 **드물다고 가정**하고, 실제 업데이트 시점에 데이터 변경 여부를 검증하는 방식.

### 구현 방식

보통 `version` 컬럼을 사용

```sql
UPDATE product
SET stock= stock-1, version= version+1
WHERE id=1AND version=3;
```

→ 영향 받은 row가 0개면 충돌 발생

### 특징

- 조회 시에는 락을 걸지 않음
- 업데이트 시점에만 버전 비교
- 게시글 수정, 사용자 프로필 수정 등 **동시 수정 가능성이 낮은 경우**에 적합

### 장점

- DB 락을 거의 사용하지 않아 성능이 좋음
- 데드락 발생 가능성이 낮음
- 읽기 위주 시스템에 유리

### 단점

- 충돌 발생 시 애플리케이션에서 재시도 로직 필요
- 충돌이 잦은 환경에서는 오히려 비효율적

---

## 3. 분산 락 (Distributed Lock)

여러 서버 인스턴스 간의 동시 접근을 제어하기 위해 **외부 저장소**를 이용하는 락 방식.

### 사용 기술

- Redis (Redlock 알고리즘)
- ZooKeeper
- Etcd

### 특징

- DB 외부에 "락 소유권"을 기록
- 여러 애플리케이션 서버 간 동기화 가능
- 결제 처리, 배치 작업 중복 실행 방지, 여러 서버에서 하나의 작업만 실행되어야 할 경우에 적합

### 장점

- 여러 DB 또는 외부 API까지 포함된 작업 제어 가능
- 마이크로서비스 환경에서 유용

### 단점

- Redis 통신 오버헤드 발생
- 별도 인프라 운영 비용 및 장애 포인트 증가
- 잘못 구현하면 락 유실 문제 발생 가능

---

## 4. 네임드 락 (Named Lock)

DB에서 문자열 키 기반으로 제공하는 **애플리케이션 레벨 락**

### 구현 방식

```sql
SELECT GET_LOCK('order_lock_123',10);
SELECT RELEASE_LOCK('order_lock_123');
```

### 특징

- 특정 문자열 이름을 기준으로 락 관리
- 메타데이터 기반 락

### 장점

- Redis 없이도 분산 락과 유사한 효과 가능
- 구현이 단순

### 단점

- DB 커넥션이 끊기면 락이 자동 해제될 수 있음
- DB 부하 증가 가능
- 대규모 분산 환경에는 한계

### 적합한 경우

- 소규모 서비스에서 간단한 분산 락이 필요할 때

---

## 5. 뮤텍스 / 세마포어 (Mutex / Semaphore)

애플리케이션 **프로세스 내부**에서 사용하는 동기화 기법

| 구분 | 설명 |
| --- | --- |
| Mutex | 하나의 스레드(고루틴)만 자원 접근 가능 |
| Semaphore | N개까지 동시 접근 허용 |

### 사용 예

- Go의 `sync.Mutex`
- Java의 `synchronized`, `Semaphore`

### 특징

- 단일 서버 내부 동시성 제어에만 유효
- 여러 서버 간 동기화는 불가능

---

## 6. 스핀 락 (Spin Lock)

락이 해제될 때까지 **반복문으로 계속 확인(busy waiting)** 하는 방식

### 특징

- 스레드가 sleep하지 않고 CPU를 계속 점유
- 매우 짧은 락에만 적합
- 커널 레벨 또는 초단기 임계 구역에 적합

### 단점

- CPU 자원 낭비 심함
- 락 보유 시간이 길면 시스템 성능 급격히 저하
