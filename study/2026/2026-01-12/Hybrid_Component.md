# Hybrid Component

## 렌더링 전략 조합

### ✅ SSR + CSR 하이브리드

- **초기 진입**
    - 서버 컴포넌트에서 **SSR(Server-Side Rendering)** 로 페이지와 초기 데이터 렌더링
    - 빠른 첫 화면 표시 및 SEO 확보
- **이후 사용자 상호작용**
    - 필터링, 페이징, 정렬 등은 **클라이언트 컴포넌트에서 CSR(Client-Side Rendering)** 로 처리
- 장점
    - 초기 UX와 인터랙션 성능을 모두 만족하는 현실적인 구조

---

### ✅ ISR (Incremental Static Regeneration)

- 목록 페이지를 **SSG(정적 생성)** 로 빌드해 매우 빠르게 제공
- 설정된 주기(`revalidate`)마다 **백그라운드에서 정적 페이지를 재생성**
- 장점
    - CDN 캐싱의 속도 + 일정 수준의 데이터 최신성 확보
- 단점
    - 실시간 데이터에는 부적합

---

### ✅ Partial Prerendering / Streaming (부분 사전 렌더링)

- **정적 셸(UI 골격)** 은 즉시 제공
- **동적 데이터 영역** 은 서버 컴포넌트 스트리밍으로 순차 렌더링
- React `Suspense` 기반
- 장점
    - 사용자에게 “빠르게 로딩되는 느낌”을 제공
    - 대규모 데이터 페이지에 유리

---

## 클라이언트 컴포넌트 (Client Component)

### 특징

- 브라우저(클라이언트)에서 실행됨
- 페이지 로드 후:
    - 클라이언트 컴포넌트의 자바스크립트 코드가 다운로드 → 파싱 → 실행
- **번들 사이즈 증가**
    - 코드가 브라우저로 전달되므로 많아질수록 초기 로딩이 느려질 수 있음
- `use client` 지시어 필요
    - App Router에서는 **기본이 서버 컴포넌트**
    - `use client` 를 선언한 컴포넌트만 클라이언트 컴포넌트로 동작
- 사용 예
    - 상태 관리 (폼, 버튼, 드롭다운)
    - 이벤트 핸들링
    - 애니메이션
    - 브라우저 API 사용 (`localStorage`, `window`, `geolocation` 등)

---

### 렌더링 흐름 (CSR 관점)

1. 사용자가 페이지 요청
2. 서버가 **최소 HTML + 클라이언트 컴포넌트 JS 번들** 전송
3. 브라우저가 JS 다운로드 및 실행
4. JS 실행이 끝난 후 UI가 완성되고 사용자 상호작용 가능
5. JS 용량이 크거나 네트워크가 느리면 **TTI(Time To Interactive)** 가 지연될 수 있음

---

## 서버 컴포넌트 (Server Component)

### 특징

- Next.js 서버 환경에서만 실행
- HTML을 서버에서 생성하여 브라우저로 전송
- **초기 로딩 속도 및 SEO에 매우 유리**
- 서버 전용 리소스 직접 접근 가능
    - DB 쿼리
    - 파일 시스템
    - API Key, Secret 환경 변수
- 보안성 우수
    - 민감한 로직이나 정보가 클라이언트로 노출되지 않음
- 컴포넌트 JS가 브라우저로 전송되지 않음
    - 결과적으로 **클라이언트 번들 사이즈 감소**
- 기본 렌더링 방식
    - SSR
    - SSG
    - ISR (SSG의 확장)

---

### SSR (Server-Side Rendering) 흐름

1. 사용자가 페이지 요청
2. Next.js 서버가 해당 경로의 **서버 컴포넌트 실행**
3. 최신 데이터를 기반으로 HTML 생성
4. 생성된 HTML을 브라우저로 전송
5. 브라우저는 HTML을 즉시 파싱해 화면에 표시
    
    → **JS 다운로드 없이도 초기 화면 확인 가능**
    

**적합한 경우**

- 사용자별 데이터
- 실시간 데이터
- 로그인 상태에 따라 다른 화면이 필요한 페이지

---

### SSG (Static Site Generation) 흐름

1. `next build` 실행
2. 빌드 시점에 서버 컴포넌트 실행
3. HTML을 **미리 생성(pre-render)** 하여 파일로 저장
4. HTML + JS + CSS 정적 파일을 CDN에 배포
5. 사용자가 요청하면 CDN이 즉시 HTML 전달
6. 브라우저는 받은 HTML을 바로 렌더링

**특징**

- SSR보다 일반적으로 더 빠름
- 서버 부하 거의 없음
- 데이터 변경이 잦은 페이지에는 부적합
