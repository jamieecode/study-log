# GPT-5.4-mini vs GPT-5.4-nano

### **모델별 특징**

|  | **GPT-5.4-mini** | **GPT-5.4-nano** |
| --- | --- | --- |
| 모델 크기 | 중형 | 초소형 |
| 핵심 강점 | 고성능 추론, 정교함 | 저지연, 가성비 |
| 멀티 모달 이해 | 정교함(OCR, 복잡한 시각 추론) | 기본 |
| 추론 능력 | 높음 | 낮음 ~ 보통 |
| 응답 품질 | 안정적 | 짧고 단순한 응답에 최적 |
| 지연 시간 | 빠름(GPT-5-mini대비 2배 이상) | 매우 빠름(실시간 인터랙션) |
| 비용 | 중간 | 매우 저렴(mini 대비 2~5배이상) |

### **멀티모달 성능**

- **공간적 이해 및 디테일:** mini는 이미지 내의 작은 텍스트(OCR), 미세한 UI 요소, 복잡한 차트의 수치를 정확히 읽어냄
- **복합 추론:** 이미지 속 상황을 보고 논리적 모순점 찾기(ex. 이미지에 표시된 합계 금액이 실제 계산 결과와 다르거나, 장면 속 환경과 시간 정보가 서로 어긋나는 경우처럼 시각 정보와 텍스트, 그리고 상식을 종합해 오류를 찾아내는 작업)와 같은 텍스트-이미지 간 교차 추론은 mini에서 훨씬 안정적으로 작동

---

### **토큰 및 비용**

- **비용 구성:** Input Tokens + Output Tokens + Image Tokens(멀티모달 사용시)
- **특이사항:**
    - Output Token 단가가 Input보다 비싸므로, 응답이 길어질수록 비용 차이가 커짐
    - Image Token은 이미지 해상도와 디테일에 따라 소모량이 달라지며, 대량의 이미지 처리 시 nano의 비용 절감 효과가 매우 큼
- **비용 전략:** 단순 분류나 태깅 등 짧은 응답이 필요한 멀티 모달 작업은 nano가 유리하며, 상세한 분석 보고서 형태는 mini가 적합

---

### **초기 개발 단계 예시**

**1단계 - nano 기반 프로토타이핑**

- **목적:** API 연동 로직 구현, 프롬프트 구조화, 응답 파싱 테스트
- **장점:** 호출 횟수가 많아도 비용 부담이 적고, 응답이 빨라 개발 생산성이 극대화됨
- **대상:** 기본 Q&A, 단순 요약, 데이터 추출 로직 설계

**2단계 - mini 기반 품질 검증 및 튜닝**

- **목적:** nano가 실패하는 복잡한 케이스 해결, 멀티 모달 정밀도 향상
- **방법:** 동일 프롬프트를 mini에 투입하여 응답 품질 비교
- **대상:** 복잡한 시각 분석, 다단계 추론, 최종 사용자 노출용 기능

### **기능별 추천 모델**

| **기능** | **추천 모델** | **이유** |
| --- | --- | --- |
| 단순 이미지 분류/태깅 | nano | 저비용으로 대량 처리 가능 |
| 문서/이미지 요약 | nano | 핵심 맥락 파악은 nano로 충분 |
| UI/UX 분석 및 스크린샷 해석 | mini | 미세한 요소 판별 및 위치 정보 중요 |
| 복잡한 차트/그래프 분석 | mini | 수치 데이터의 정확성 및 추론 필요 |
| 코드 리뷰 및 로직 생성 | mini | 높은 논리적 완결성 요구 |
| 실시간 챗봇 인터페이스 | nano | 사용자 경험을 위한 즉각적 응답 속도 |

### **개발 시 주의사항**

- **초기 단계에서는 무조건 mini 고정 사용 x:** 초기 실험 단계에서 수천 건의 API 호출 시 불필요한 비용이 발생
- **비용 시뮬레이션:** 멀티 모달은 텍스트보다 토큰 소모가 크므로, 반드시 이미지 토큰 비용을 계산할 것
- **단일 모델 아키텍처:** 모든 기능을 한 모델에 맡기지 말고, 난이도에 따라 **모델 라우팅(Routing)** 로직을 검토

---

### **주요 라우팅 방법**

라우팅 로직이 너무 복잡하면 오히려 Latency가 발생할 수 있음. 최대한 단순한 규칙으로 시작할 것

- **키워드 및 의도 기반 라우팅**
    - 가장 구현 쉬움. 사용자의 질문에 특정 키워드가 있거나 미리 정의된 카테고리에 해당하면 모델 선택
- **분류기 모델(Classifier-based)**
    - 매우 가벼운 별도의 분류 모델을 사용해 요청의 난이도를 먼저 판단
    - 작동 방식
        - 사용자 요청 → 분류기(LLM)가 난이도 판단 → 5점 이하면 nano, 6이상이면 mini 호출
- **멀티 모달 데이터 유무**
    - 입력 데이터 형식에 따라 나누기
        - 이미지 파일 포함: mini (정확도)
        - 텍스트만 포함 시: nano (속도)
- **폴백 전략**
    - 먼저 저렴한 모델로 시도 → 결과에 따라 상위 모델로 다시 시도
    - 작동 방식
        - nano에게 먼저 시킴 → 응답이 “모르겠습니다” 이거나 형식이 맞지 않으면 자동으로 mini에게 재요청하도록 구현
