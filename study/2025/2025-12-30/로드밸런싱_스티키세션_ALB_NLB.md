# 로드밸런싱/스티키세션/ALB/NLB

## 로드 밸런싱(Load Balancing)

- 트래픽이 많을 때 여러 대의 서버가 분산처리하여 서버의 로드율을 증가시켜 주고, 부하량이나 속도 저하등을 고려해 분산 처리해주는 서비스
- **동작 방식**
    - 네트워크 상단에 가상서버가 존재하여 서버로 들어오는 패킷을 리얼 서버로 균일하게 트래픽을 분산시킴
    - 서버에 장애가 발생하면 이를 감지해 정상 작동하는 서버로 트래픽을 분산
- **장점**
    - 한 서버가 다운되더라도 이중화시킨 다른 서버에서 서비스 지속하여 서비스들이 문제를 인지하지 못하게 할 수 있음
    - 노드를 추가하는 것만으로 서비스가 확장성을 가질 수 있음
- **단점**
    - 세션 데이터 관리가 어려움
    - 클라이언트 연결정보를 저장하는 세션이 로드밸런싱을 통해 하나의 서버 장비에 저장되는 경우, 추후 다른 서버로 접속하게 되면 해당 클라이언트의 세션이 유지되지 않음→ 이 문제를 해결하기 위해 스티키 세션 사용
- **로드 밸런서**(Load Balancer): 로드 밸런싱을 해주는 소프트웨어/하드웨어 장비

---

## 스티키 세션(Sticky Session)

- **로드밸런서가 세션 기간 동안 동일한 클라이언트의 요청을 항상 동일한 서버로 라우팅 해주는 기능**
- 유저가 첫번째 세션을 생성한 서버로 모든 요청을 리다이렉트하여 고정된 세션만 사용하게함
    
    ex. 유저1이 1번 서버에 세션 생성 → 이후에 유저1이 보내는 모든 요청은 1번 서버로만 보내짐
    
- **로드밸런서는 요청을 받으면 가장 먼저 요청에 쿠키 존재 여부 확인**
    - 쿠키가 있으면 해당 요청이 쿠키에 지정된 서버로 전송됨
    - 쿠키가 없으면 로드밸런서가 기존 로드 밸런싱 알고리즘 기반으로 서버를 선정
- **장점**
    - 서버들 간 세션 데이터 교환할 필요 없음
    - 정합성 이슈x
- **단점**
    - 특정 서버 과부하 발생 가능. 트래픽 균등 배분 어려움
    - 로드밸런싱으로 트래픽이 분산되긴 하지만, 스티키세션을 사용했을 때 특정 서버에 몰린 사람들만 활발하게 활동하는 경우 해당 서버만 과부하 걸릴 수 있음
        - 특정 서버에 과부하 걸리면 로드 밸런서는 이를 감지하고 그 서버로 향하는 트래픽을 다른 서버로 다시 라우팅하는데 이 과정에서 기존 세션 데이터가 유실됨→유저 재로그인 필요

## ALB(Application Load Balancer)

- 스티키 세션(Sticky Session) 지원
    - 로그인한 사용자가 같은 서버에 계속 붙어있을 수 있음
    - 분산 기능 없음, 자동 확장 불가(세션 유지에 초점)
    - 스티키니스를 활성화하면 백엔드 EC2 인스턴스의 부하에 불균형이 생길 수 있다. (스티키 세션이 동일한 클라이언트가 항상 로드 밸런서 뒤에 있는 동일한 인스턴스로 리디렉션되도록 돕기 때문)
- HTTPS(SSL/TLS) 지원
    - ACM 인증서 붙여서 보안 연결 처리 가능
- **L7 계층(애플리케이션 계층)**
    - HTTP, HTTPS 요청 내용을 보고 판단(URL, 호스트, 헤더 등등)
- URL/도메인 기반 라우팅(ex.  A서버: /login, B서버: /payment)
- 컨테이너 지원(ECS 등)
    - 하나의 인스턴스에서 여러 포트로 컨테이너 분산 가능
- 상태확인(헬스 체크)
    - 죽은 서버는 자동으로 제외하고 살아있는 서버에만 트래픽 보냄
- WAF, CloudWatch 연동 가능
    - 보안, 모니터링 굳
- **고정 IP는 없음**
    - NLB처럼 고정된 IP는 없지만 DNS로 접근 가능

---

## NLB(Network Load Balancer)

- **고정 IP 제공 및 탄력적 IP(EIP)와 결합 가능**
    - 클라이언트에게 보여줄 수 있는 고정된 IP주소 제공
- TLS 종료 지원
    - 보안 연결(SSL/TLS) 처리 가능
    - 서버 인증서 붙이면 HTTPS 통신도 NLB에서 처리 가능
- 고성능, 대규모 트래픽 처리에 유리
- **L4 계층(네트워크 계층)**
    - TCP, UDP 요청 빠르게 분산시킴(내용 안봄)
- 초고속 처리
    - 초당 수백만개 요청 처리 가능, 지연 거의 없음
- 정상 대상만 트래픽 전달
    - 헬스체크 기능으로 죽은 서버 자동 제외
- 트래픽에 따라 자동 확장, 크기 조절 가능
- 가용 영역 여러 개로 분산 가능
    - 장애에 강함. AZ별 분산, Cross-zone 설정 가능
- 클라이언트 IP 보존
    - 서버는 요청 보낸 사람의 IP 주소를 직접 확인 가능
